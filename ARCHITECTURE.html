Flash Sandbox Engine

A self-hosted engine that hands out isolated Docker sandboxes from warm pools — claimable in under 2 seconds via API key + SDK. A generic /v1/sandboxes API, a native dashboard baked into the binary, and an assessment layer (claim → submit → automated scoring) on top.

01 The whole system in one picture

Everything the product needs lives in one Go binary: the HTTP API, the auth layer, the in-process warm pool, the WebSocket terminal proxy, the preview reverse-proxy, the scorer, and the embedded dashboard. External state is just Postgres (durable rows) and Docker (the sandboxes themselves).

No SQS, no Redis (single-node), no ElastiCache. Pool state is a mutex-guarded map — correct for one node.
Design spine: the control plane never talks to Docker directly — it talks to a Provisioner interface. The local impl is an in-process pool; the cluster impl schedules across node-agent boxes. Same control-plane code, two backends (see §08).

02 Two planes, three audiences

The same binary serves three surfaces, each authorized differently. Keeping the auth model straight is the key to reading the rest of the architecture.

Operators hold org keys; candidates hold single-session tokens; the toolbox is never exposed publicly.

03 The sandbox lifecycle

One request path drives the whole product: claim a pre-warmed container, hand back live URLs, let the user work, then submit → score → destroy → replenish. The magic is that the container is already running when the claim arrives.

Claim is fast because provisioning already happened; scoring is safe because it runs network-isolated and one-shot.

Measured locally: starter code scores 25/100, a correct solution 100/100, and the warm pool self-heals back to its min_warm floor after each destroy.

04 The warm pool state machine

The pool (internal/pool) is the engine's heart: a mutex-guarded map that keeps min_warm containers spun-up and healthy per template, so a claim is a cheap pop rather than a cold docker run. It also owns port allocation, health checks, timeout reaping, and adoption of survivors after a restart.

A claim moves a box WARM → ACTIVE instantly; a background loop keeps refilling the WARM band to min_warm.
Restart safety: ACTIVE sessions are durable rows in Postgres. On boot the pool adopt()s still-running containers that a live session points at, so an orchestrator restart doesn't orphan an in-progress candidate.

05 Live terminal — the WebSocket proxy chain

A token-authed WebSocket gives the candidate a real shell in /app/src. The orchestrator is a transparent frame-forwarder between the browser and the toolbox's PTY — it never interprets terminal bytes, so resize and control sequences work end-to-end.

Two hops, one verbatim relay: GET /v1/sessions/:id/terminal?token= upstream, /ws/terminal to the toolbox PTY.

06 Browser preview — wildcard-subdomain reverse proxy

Frontend templates (e.g. q3 React/Vite) run a live dev server the candidate sees in a browser. The orchestrator fronts it on a single authenticated entrypoint using wildcard subdomains — the Caddy + wildcard-DNS shape, but with zero DNS setup (any *.localhost resolves to loopback per RFC 6761).

One authenticated origin per session; the cookie handshake keeps the token out of every subsequent request.

07 Submit & score — isolated, one-shot, deterministic

Submission is the assessment payoff. The candidate's source is snapshotted, then a fresh scoring container runs the template's harness with --network none — no exfiltration, no external calls — and emits a JSON verdict that lands in Postgres. Then the working container is destroyed.

Scoring runs in its own network-isolated container so untrusted candidate code can't reach the harness's world.

08 Single node → fleet: the concurrency unlock

The Provisioner interface is the seam that lets one box become many without touching control-plane code. Unset CLUSTER_MODE = an in-process pool (v1). Set it + REDIS_URL = a stateless control plane scheduling across node-agent boxes.

Scheduler prefers a node already warm for the template, else bin-packs onto the box with the most free memory.

Routing

Preview / terminal proxies and submit/score route to the owning node via the durable session row in Postgres.

Failover

Killing a node reschedules new claims onto the survivor (proven 8/8). An ungraceful death loses that node's in-flight sessions.

Graceful drain

SIGTERM → refuse new claims, fail /readyz, finish active sessions, then exit (11/11).

09 Scheduled scaling & the autoscaling signal

A booked assessment shouldn't start cold. The planner raises a template's warm floor lead minutes ahead of a window and restores it after — so a 10:00 assessment is warm at 09:55. It also publishes the signal an ASG scales the fleet on.

Warm capacity is pre-provisioned to match booked demand, and the required-nodes signal drives the underlying fleet size.

10 Data model & package map

Postgres holds only what must be durable: identity, keys, sessions (with node routing), and submissions. Everything ephemeral (which container is warm right now) lives in memory / Redis.

Durable rows only — ephemeral pool state stays in memory / Redis.
